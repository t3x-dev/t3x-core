How should frames render in commit views?

This pattern will be the foundation for commit detail, diff, and merge pages. Frames have a type, key-value slots, and optional relations.

f_1 · travel_plan · 0.92

- cooking classes
- tea ceremony

A. YAML Cards

Each frame is a code-like YAML card with syntax highlighting. Type + confidence as header. Slots as key: value pairs. Hover a slot to highlight source text.

Pros

  • Native format - "YAML is the product"
  • Information dense
  • Consistent with extraction panel

Cons

  • Can feel technical
  • Nested values need careful formatting

B. Structured Cards

Clean UI cards with type badge, key-value grid layout, and pill tags for arrays. More "app-like" than code-like.

Pros

  • Polished, approachable
  • Good for non-technical users
  • Rich layout for nested data

Cons

  • Diverges from extraction panel style
  • Less information density
  • Custom rendering per slot type

C. Hybrid (Recommended)

YAML body for information density + structured header bar with type badge, ID, and confidence. Best of both worlds — keeps the "YAML is the product" principle while adding visual polish.

Pros

  • YAML body = consistent with extraction panel
  • Structured header = scannable metadata
  • Naturally extends to diff (color YAML lines)

Cons

  • Slightly more complex to implement
