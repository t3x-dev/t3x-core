# T3X MCP E2E 测试场景讨论稿

## 文档目的

这份文档用于团队讨论 `MCP 跑通之后，优先验证哪些真实用户使用场景`。

这里的重点不是验证单个 tool 能不能调用，而是验证：

- 用户是否能通过 AI 客户端自然地使用 T3X
- MCP 是否能支撑完整的产品任务闭环
- T3X 最核心的语义版本控制能力是否在真实场景里成立
- MCP 返回的结果是否足够让用户继续下一步，而不是停在一段原始 JSON

本文先聚焦 5 个高价值场景，供团队过目、取舍和排序。

## 场景选择原则

本次 E2E 场景选择遵循 4 个原则：

1. 场景必须接近真实客户任务，而不是为了测工具而造工具。
2. 场景必须覆盖 T3X 的核心价值，即提取、演进、对比、合并、产出。
3. 场景应尽量让团队看懂“用户为什么会用”，而不是只看见 MCP 调用序列。
4. 场景要能在 `MCP 跑通` 后分阶段落地，先做核心链路，再补展示和细节。

## 推荐的 5 个真实用户场景

### 场景一：写项目计划书

#### 用户画像

项目负责人、产品经理、咨询顾问。

#### 用户任务

用户把多轮讨论记录、会议纪要、需求补充发给 AI，希望 AI 帮他整理成一份结构化的项目计划基础稿，包括目标、范围、里程碑、预算、风险、待确认事项等。

#### 为什么适合 T3X

这是典型的“从非结构化对话提取稳定知识”的任务。用户不是只要一段总结，而是希望后续还能持续修改和迭代。

#### 推荐的 MCP 闭环

`create_project -> extract -> query(draft) -> edit(必要时) -> commit`

后续如果已经有可查询的 `leaf_id`，可以再接 `generate`。

#### E2E 验证重点

- 多轮对话能否成功抽取成结构化 draft
- draft 中是否形成计划书需要的核心节点
- 用户补充一两条要求后，draft 是否能稳定更新
- commit 是否能形成可追踪的知识快照
- MCP 返回的信息是否足够让用户知道“下一步可以继续 refine 或生成文稿”

#### 通过标准

用户能在一个自然会话中，从原始讨论走到“有一个可持续演进的计划知识底稿”。

### 场景二：写决策记录

#### 用户画像

产品负责人、技术负责人、管理层助理。

#### 用户任务

用户围绕一个方案做讨论，例如“是否自研”“是否延期”“是否改技术栈”，最后希望沉淀出一份决策记录，包含背景、备选方案、关键依据、最终结论、风险和待跟进项。

#### 为什么适合 T3X

决策不是静态文本，而是会随新信息不断变化。T3X 的价值在于把“决策内容”沉淀为可比较、可演进、可回看的语义版本。

#### 推荐的 MCP 闭环

`create_project -> extract -> query(draft) -> edit -> commit`

如果后续又有新信息，再走一轮 `extract -> query(draft) -> edit -> commit`。
如果一定要覆盖 `generate`，要明确前置条件：当前项目里已经存在可查询的 `leaf_id`。

#### E2E 验证重点

- 能否从讨论中提取出决策相关的结构化信息
- 用户新增约束后，是否能生成新的 commit
- 新旧 commit 之间是否可以清晰追踪决策变化
- 生成出来的决策记录是否确实建立在 commit 知识之上
- 输出中是否保留足够的可解释性，便于回看

#### 通过标准

用户不仅能得到“当前版决策记录”，还能够看到决策是如何演化出来的。

### 场景三：需求评审后连续迭代方案

#### 用户画像

产品经理、设计师、研发负责人。

#### 用户任务

用户在需求评审会上不断增加限制条件，例如预算变化、时间变化、范围收缩、风险增加，希望 AI 不是每次重写一份文档，而是在原有语义基础上持续修改。

#### 为什么适合 T3X

这类任务最能体现 T3X 和普通聊天工具的区别。真正有价值的是“在已有结构上做增量变更”，而不是每次都重新总结。

#### 推荐的 MCP 闭环

第一轮：`create_project -> extract -> query(draft) -> edit(必要时) -> commit`

后续多轮：`extract(project_id, text, conversation_id) -> query(draft) -> edit(必要时) -> commit`

#### E2E 验证重点

- query 返回的信息是否足够让 agent 理解当前状态
- edit 是否能只修改相关节点，而不是破坏整棵树
- 多轮 revision 过程中，draft revision 是否正常推进
- 连续多个 commit 后，语义内容是否仍然稳定、可读、可解释

#### 通过标准

用户在连续数轮补充和修正后，仍能得到一条清晰的演进链，而不是多份彼此割裂的草稿。

### 场景四：比较两个版本的方案差异

#### 用户画像

项目负责人、销售方案经理、解决方案架构师。

#### 用户任务

用户需要比较两个版本的方案，例如“初版计划”和“评审后计划”、“客户 A 版本”和“客户 B 版本”，希望 AI 能告诉他到底哪些内容变了、哪些是新增的、哪些被删除了。

#### 为什么适合 T3X

这是 T3X 最核心的价值表达之一。用户真正关心的不是文件 diff，而是语义 diff。

#### 推荐的 MCP 闭环

`extract -> query(draft) -> edit(必要时) -> commit v1 -> extract(new input) -> query(draft) -> edit(必要时) -> commit v2 -> diff(v1, v2)`

如果要展示给用户看，MCP 返回结果后还可以补一个详情链接进入 Web 查看。

#### E2E 验证重点

- 通过 YAML 编辑或新输入生成多个 commit 是否稳定
- diff 是否能正确区分 identical、modified、only in source、only in target
- slot 变化、节点变化、关系变化是否都能被识别
- MCP 返回的 diff 结果是否足够让用户直接消费，而不需要人工二次翻译

#### 通过标准

用户可以清楚回答“第二版到底和第一版差在哪里”，而不是只能看到一堆底层结构变化。

### 场景五：从多个意见分支收敛成统一结论

#### 用户画像

跨团队负责人、产品委员会、技术委员会。

#### 用户任务

同一个主题上存在多个方向，例如业务方要快上线、技术方要控风险、运营方要顾体验。用户希望 AI 先把不同意见沉淀为多个语义版本，再帮助准备合并，最终形成一个统一方案。

#### 为什么适合 T3X

如果只做提取和总结，这类分歧场景很难体现产品优势。只有进入 `prepare merge -> resolve -> execute`，团队才会真正看到 T3X 的差异化价值。

#### 推荐的 MCP 闭环

`commit source -> commit target -> merge.prepare -> merge.show_conflict -> merge.resolve -> merge.execute`

如果需要看 merge draft 的完整结构，当前更适合通过资源读取，而不是写成普通 `query` 闭环。

#### E2E 验证重点

- merge prepare 是否能把自动保留项和冲突项正确拆开
- 冲突展示是否足够让 agent 或用户做判断
- resolve 后 execute 是否能生成合法 merge commit
- merge commit 是否正确保留父提交关系
- 最终结果是否能表达“不同意见如何被吸收”

#### 通过标准

用户能把“多方意见”从对立状态推进到一个可审阅、可追踪的统一语义版本。

## 为什么这 5 个场景值得优先做

这 5 个场景组合起来，基本覆盖了用户使用 T3X MCP 的一条完整价值链：

- 计划书和决策记录，验证“知识能进来”
- 连续迭代，验证“知识能演进”
- 版本比较，验证“变化能看见”
- 多分支合并，验证“分歧能收敛”

如果这些场景跑通，团队就能更有把握地对外说明：

- T3X 不是一个只会总结对话的 MCP
- T3X 可以把对话沉淀为可版本化、可比较、可合并的知识对象
- 用户通过 AI 客户端使用时，能够完成真实工作，而不是只得到一段临时回答

## 建议的优先级

### P0：必须先做

1. 写项目计划书
2. 需求评审后连续迭代方案
3. 比较两个版本的方案差异

### P1：建议尽快补上

4. 写决策记录
5. 从多个意见分支收敛成统一结论

原因很直接：

- `计划书` 是最自然的入口场景
- `连续迭代` 最能体现不是一次性总结
- `版本 diff` 最能体现 T3X 的核心辨识度
- `决策记录` 和 `merge` 更适合在核心链路稳定后再做强验证

## 团队讨论时建议重点确认的问题

1. 这 5 个场景里，哪些最接近我们第一批目标用户的真实任务？
2. 我们要优先证明“能跑通”，还是优先证明“有产品价值”？
3. MCP 返回给用户时，是先以结构化文本为主，还是同步补 Web 详情入口？
4. 哪些场景必须覆盖 `generate`，哪些场景只需要走到 `commit / diff / merge` 就足够？
5. E2E 的通过标准，是偏底层正确性，还是偏用户能否完成任务？

## 当前建议

如果明天团队只想先收敛一个最小可讨论范围，我建议先围绕下面三条推进：

1. 写项目计划书
2. 需求评审后连续迭代方案
3. 比较两个版本的方案差异

这三条既贴近用户任务，也能直接覆盖 `extract / query / edit / commit / diff` 这条主链路。

后续等主链路稳定，再补：

4. 写决策记录
5. 多分支 merge

这样节奏更稳，也更容易让团队先形成共识。
