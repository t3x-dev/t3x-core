## 1. 三条基因线

T3X 的参考产品可以归为三条基因线。策略是：从每条线各取一个最关键的机制，拼成一个闭环产品。

| 基因线 | 产品 | 共同点 |
|--------|------|--------|
| **A. 协作** | GitHub、Figma、Notion | 多人在同一个"资产"上协作，协作成本极低 |
| **B. 工程** | dbt、Dagster、n8n、Vercel | 把能力"运营化"：能跑、能看、能复现、能上线、能回滚 |
| **C. 生态** | LangChain | 标准接口 + 组件生态 |

## 2. 基因线 A：协作型产品

### 2.1 GitHub —— 该学什么

**核心机制**：Issue/PR = 结构化协作输入；Review/Checks = 合并前的质量门槛。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **Commit PR** | 每次把 commit/leaf 推到主线，都需要：变更摘要、证据、影响面、eval 结果（如有 Runner） |
| **Merge Checks** | 自动化质量门：约束是否满足、哈希是否一致、eval 是否通过、冲突是否处理完 |

### 2.2 Figma —— 该学什么

**核心机制**：Branch = 安全试错空间（是产品概念，不是工程概念）；Review = 可视化的、可理解的差异。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **分支即实验** | UI 上把"branch"改成"实验/变体/草稿"语言；底层引擎保留 `branch` |
| **人话差异摘要** | 合并面板展示：新增/删除/替换了哪些语义块、哪些关键句被保留/被替换、为什么 |

### 2.3 Notion —— 该学什么

**核心机制**：Template Gallery + 一键复制 + 分享链接 = 增长引擎。内容资产可传播、可复用。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **Leaf 模板库** | 必须做。开源爆红的核心驱动力。首发 10-20 个行业模板。 |
| **一键分享** | 每个 Leaf、Context Packet、Eval Report 都要能生成可分享的链接（哪怕只读） |

## 3. 基因线 B：工程型产品

### 3.1 dbt —— 该学什么

**核心机制**："资产化"思维。模型是资产、文档是资产、血缘是资产。Cloud 负责可协作、可观测、可规模化。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **T3X 资产** | Commit、Leaf、Context Packet、RunTrace、EvalSuite——每一个都是一等资产 |
| **资产元数据** | 每个资产有：状态、血缘（谁生成的、引用了哪些来源）、可追溯链路（hash/commit chain） |

### 3.2 Dagster —— 该学什么

**核心机制**：Asset Graph + Materialization（每次运行产出一次"可记录的物化结果"）。可观测性：每次运行展示输入→步骤→输出。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **物化报告** | Runner 结果不只是日志，要变成可复用、可比较、可分享的报告 |
| **运行透明度** | 用户能看到：这次 eval 用了哪些 commit、哪套规则、产生哪些指标、和上次的差异 |

### 3.3 n8n —— 该学什么

**核心机制**：低代码工作流 + 节点生态 + Webhook。n8n 的强项不是功能，是"连接能力"——让别人把 T3X 接进自己的系统。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **工作流配方包** | 可配置的自动化：监听 PR/issue → 自动生成 commit → 跑 Runner → 生成报告 → 发 Slack/Notion |
| **第一阶段策略** | 不做可视化工作流编辑器，先做可配置的"配方 + webhook 触发"就能产生生态效应 |

### 3.4 Vercel —— 该学什么

**核心机制**：Preview Deployments（每个改动都有一个可分享的预览）。从 commit 到可见结果的最短路径决定传播速度。

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **预览链接** | 每个 Leaf 和 Context Packet 都有可分享的预览链接（类似 Vercel 的预览部署） |
| **自动生成发布说明** | 每次合并后自动生成"Release Note / Diff Summary"——可传播内容 |

## 4. 基因线 C：生态与接口层

### 4.1 LangChain —— 该学什么、不该学什么

**核心机制**：统一抽象层（providers/tools/runners/connectors）+ 生态贡献机制。

| 该学 | 不该学 |
|------|--------|
| 跨 providers/tools/runners/connectors 的统一抽象层 | 把核心做成"随便拼"的积木，导致产品失去确定性和可审计性 |
| 生态贡献机制（别人能贡献组件） | 为了灵活性牺牲确定性 |

T3X 需要的：

| 概念 | T3X 落地方式 |
|------|-------------|
| **确定性核心** | T3X 的优势是确定性（commit/hash/diff/merge），这是底盘，不能被"灵活性"稀释 |
| **生态边界** | 生态放在 Leaf/Runner/Integrations，永远不进 Core |

## 5. 组合定位

T3X 的最佳组合拳：

```
GitHub（协作治理）
+ Notion（模板 & 传播）
+ Dagster/dbt（资产化 & 可观测）
+ n8n（连接 & 自动化）
+ Vercel（预览 & 分享）

LangChain 做"插件/生态层"，但不做"核心形态"。
```

一句话定位：

> T3X 是一个确定性语义版本控制系统，让 AI 对话知识可治理（GitHub）、可传播（Notion）、可观测（Dagster）、可连接（n8n）、可预览（Vercel）。

## 6. 里程碑路线图

目标：开源发布时实现最大采用冲击力。

### Milestone A：Notion 化（可复制、可传播）

| 交付物 | 详情 |
|--------|------|
| Leaf 模板库 | 10-20 个行业模板，可搜索/可筛选 |
| 一键使用 | "使用此模板" 即刻创建预填好的 Leaf |
| 分享链接 | 每个 Leaf、Context Packet、Eval Report 生成只读分享链接 |
| 导出 | 所有资产支持 Markdown、JSON、Prompt 文本导出 |

### Milestone B：GitHub 化（可协作、可审计）

| 交付物 | 详情 |
|--------|------|
| Commit PR | 推到主线需要：变更摘要、证据、影响面、eval 状态 |
| Merge Checks | 自动化质量门：约束满足、哈希一致、eval 通过、冲突处理完 |
| 合并摘要 | 人话变更概要（Layer 1 内容）：新增/删除/替换了什么、哪些关键句被保留 |
| 术语体系 | UI 全面使用默认模式词汇（快照/变体/合并），开发者模式切换支持 Git 术语 |

### Milestone C：Dagster/Vercel 化（可运行、可对比、可预览）

| 交付物 | 详情 |
|--------|------|
| 物化 Runner 报告 | 每次运行产出一等、可分享、可比较的报告资产 |
| 对比视图 | A/B 测试、版本 Diff、跨次运行差异——集成在一个视图中 |
| 预览链接 | 任何结果都能一键分享 |
| 工作流配方 | 可配置的配方 + webhook 触发（PR → commit → 运行 → 报告 → Slack） |

## 7. 与前端规则的冲突分析

与 `docs/frontend-rules.md` 的交叉对照。分析于 2026-02-09 执行。

### 完全对齐（无需修改）

| 策略要求 | 对应前端规则 |
|---------|-------------|
| 确定性核心，生态放在 Leaf/Runner | 规则 8 + 第 3.1 节 |
| 两类用户，80% 不见 Git 术语 | 规则 1（术语表已添加） |
| 溯源链可追踪 | 规则 8 |
| 空状态即教学 | 规则 5 |
| 模板库 | 第 3.4 节（已添加） |

### 架构张力点（需要团队决策）

| 张力 | 策略说 | 规则说 | 可选方案 |
|------|--------|--------|---------|
| **预览/分享链接** | 每个资产都能生成分享链接 | 第 3.2 节：所有页面都是 Client 组件 | A) API 渲染分享页（Next.js 之外） B) 新增 Server Component 分享路由 C) 全面混合渲染迁移 |
| **合并摘要** | 人话摘要放在最显眼位置 | 规则 3：Diff 在 Layer 3 | 自动生成的摘要 = Layer 1。Diff 工具 = Layer 3。两个不同概念，无实质冲突。 |
| **Commit PR 流程** | 合并是流程，不是按钮 | 当前：直接执行合并 | 需要新状态（pending_review/approved），可能需要在 Orange→Blue 之间扩展中间色。 |

### 已补入已知缺口表的项目

|------|--------|--------|
| Commit PR + Merge Checks | B | P1 |
| 合并摘要（Layer 1） | B | P1 |
| 预览链接 / 分享链接 | A/C | P1 |
| Runner 报告资产化 | C | P1 |
| 工作流配方 / Webhook 触发 | C | P2 |
